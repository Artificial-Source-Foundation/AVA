# Install And Release Paths

This page maps AVA's public install and release paths.

Use this page when you want the factual source of truth for install surfaces and release artifacts.

AVA currently has two user-facing install surfaces:

1. `ava` CLI/TUI
2. AVA Desktop

Practical reading guide:

1. If you want step-by-step instructions, use [How-to: Install AVA](../how-to/install.md) or [How-to: Download AVA Desktop](../how-to/download-desktop.md).
2. If you want the authoritative artifact and ownership map, stay on this page.

Current state: release-related files in this checkout use the `Artificial-Source/AVA` org/repo slug consistently.

## At A Glance

| Surface | Primary user path | Current source of truth |
|---|---|---|
| CLI binary install on Linux/macOS | `install.sh` | Installer script plus GitHub Releases |
| CLI binary install on Windows/manual installs | GitHub Releases assets or generated installer | GitHub Releases |
| CLI web-enabled source install | `cargo install --path /path/to/AVA/crates/ava-tui --bin ava --features web --force` | Local source build |
| CLI source install | `cargo install --path /path/to/AVA/crates/ava-tui --bin ava` | Local source build |
| Guided source build | `./install-from-source.sh` | Local source build helper |
| Desktop app release/download path | GitHub Releases when desktop bundles are published | GitHub Releases |

## CLI Binary Distribution Path

### Installer script path

The repository ships `install.sh` for Linux/macOS binary installation.

Current behavior in `install.sh`:

1. Supports `linux` and `macos`
2. Supports `x86_64` and `aarch64`
3. Resolves latest GitHub release tag
4. Downloads the newest matching CLI archive, supporting both legacy `ava-<target>.tar.gz` assets and current `ava-tui-<target>.tar.xz` assets (plus optional `.sha256`)
5. Installs to `~/.ava/bin` by default

Script usage is documented in the script header:

```bash
curl -fsSL https://raw.githubusercontent.com/Artificial-Source/AVA/develop/install.sh | sh
```

### GitHub Releases path

The release workflow is defined in `.github/workflows/release.yml` (generated by `cargo-dist`) and publishes artifacts on version tags.

The dist configuration currently enables both shell and PowerShell installers. That means the documented Unix-first `install.sh` script is not the only release-install surface: Windows/manual users should use the GitHub Releases page and select the matching asset or generated installer for the version they want.

Configured dist targets currently include:

1. `aarch64-apple-darwin`
2. `x86_64-apple-darwin`
3. `aarch64-unknown-linux-gnu`
4. `x86_64-unknown-linux-gnu`
5. `x86_64-pc-windows-msvc`

### CLI asset naming examples

The CLI release archives use target-triple naming. Current examples are:

1. `ava-tui-aarch64-apple-darwin.tar.xz`
2. `ava-tui-x86_64-apple-darwin.tar.xz`
3. `ava-tui-aarch64-unknown-linux-gnu.tar.xz`
4. `ava-tui-x86_64-unknown-linux-gnu.tar.xz`
5. `ava-tui-x86_64-pc-windows-msvc.zip` or an equivalent generated Windows installer entry on the release page

## CLI Source Install Path

The documented source install path is:

```bash
cargo install --path /path/to/AVA/crates/ava-tui --bin ava
```

For web mode, the current docs and crate feature flag indicate an explicit `web` feature build:

```bash
cargo install --path /path/to/AVA/crates/ava-tui --bin ava --features web --force
```

For source builds without installation, the repo-level command is:

```bash
cargo build --manifest-path /path/to/AVA/Cargo.toml --bin ava
```

Profile guidance:

1. `cargo build` uses Cargo's normal dev profile with debug info and faster incremental rebuilds.
2. `cargo build --release` uses the optimized release profile and is the closer match for published binaries.
3. Cargo does not provide a built-in `--debug` or `RelWithDebInfo` switch; use `--profile <name>` if the repo defines an additional profile.

## Guided Source Build Path

The repository also ships `install-from-source.sh` for users who want a more guided source build from a local checkout.

Current script scope:

1. Can install CLI-only, desktop-only, or both
2. Supports non-interactive flags such as `--cli`, `--desktop`, and `--all`
3. Installs into user-owned locations under `$HOME`
4. Defaults `CARGO_BUILD_JOBS` to detected CPU count unless the caller already set it
5. Enables `sccache` automatically when available unless the caller already set `RUSTC_WRAPPER`

## Desktop Release Path

Desktop release instructions are maintained separately.

Current documented desktop bundle outputs are:

1. Linux: `.deb`, `.AppImage`, `.rpm`
2. macOS: `.dmg`, `.app`
3. Windows: `.msi`, `.exe`

See: [Contributing: Releasing](../contributing/releasing.md)

This split is intentional: CLI binaries are distributed through the automated GitHub release path, while desktop packaging/signing is documented as an explicit maintainer process.

For end users, the practical implication is simple: when maintainers attach desktop bundles to a GitHub Release, that release page becomes the download surface. If a given release does not include the desktop bundle you need, the supported fallback is a source build from a local checkout.
